# FundedNext Futures Page — Agentic AI Handoff
*How to safely read, edit, and verify `cpf-fundednext.html`. Pair this with `fundednext-content-map.md` (the data source of truth) and the cornerstone `cpf-tradeify.html` (the immutable design reference). This page was built by swapping content into the cornerstone; structure and design tokens must not drift.*

---

## 0. Golden rules
1. **Never invent data.** Every numeric/policy value comes from `fundednext-content-map.md`, which is itself verified against helpfutures.fundednext.com. If a value isn't there, fetch the first-party article and add it to the map first.
2. **Edit with guarded scripts.** Use a Python `rep(old, new, n)` helper that asserts the exact occurrence count and aborts (without writing) on mismatch. No blind regex sweeps.
3. **Validate JS after every JS-touching change:** extract each `<script>` block and run `node --check`.
4. **Render-test interactive changes** with headless Chromium (Playwright) — assert the rendered text/colors, don't eyeball the source.
5. **Self-contained file.** All CSS is inline `<style>`, all JS inline `<script>`, images are base64 data URIs. No external build step, no localStorage. Keep it that way.
6. **Match the cornerstone.** Design tokens, type system (Inter / Source Serif), spacing, and component CSS are inherited from `cpf-tradeify.html`. Only content and the documented per-firm divergences differ. Orange = CTA/action only; teal = callouts/pills/tags.

### Standard workflow
```
cp /mnt/user-data/uploads/cpf-fundednext.html /home/claude/work/   # uploads are read-only
# ...edit in /home/claude/work via guarded Python...
python3 - <<'PY'  # node --check every script block
import re;t=open('cpf-fundednext.html').read()
[open(f'/tmp/b{i}.js','w').write(b) for i,b in enumerate(re.findall(r'<script>(.*?)</script>',t,re.S))]
PY
for f in /tmp/b*.js; do node --check "$f"; done
# ...Playwright render test...
cp cpf-fundednext.html /mnt/user-data/outputs/ ; # then present_files
```

## 1. Page architecture (DOM map)

**Top-level tabs** (`.tab[data-tab=…]` → `#panel-…`): `overview`, `accounts`, `rules`, `payouts`, `restricted`, `faq`. `FULL_WIDTH_TABS` hides the right sidebar for content-heavy tabs.

**Accounts tab has two views** (`.view-toggle-btn[data-view=…]`):
- **Comparison view** (`#view-comparison`): one `.account-card[data-account=rapid|legacy|bolt|flex]` per model. Each card has `.ac-sizes` (size pills, always show **price**) + `.ac-stats` (rows built dynamically — innerHTML is generated by JS from row schemas, not static `data-field` markup). A compact, centered **Evaluation/Funded phase toggle** (`.phase-toggle-btn[data-phase=eval|funded]`) sits above the grid and only shows in this view; Eval and Funded render **different row sets**.
- **Card view** (`#view-card`): a single "card detail" panel driven by `cardState = {type, size}`. Type buttons (`.cd-type-btn[data-cardtype=…]`), size buttons (rebuilt per type into `[data-cd-sizes]`), and seven sub-tabs (`.cd-subtab[data-cd-tab-target=…]`): `overview`, `evaluation`, `rules`, `funded`, `scaling`, `payouts`, `faq`.

**Card-detail Overview = the six boxes** (`.cd-block` with `.cd-block-title`): 🎯 Evaluation · 📈 Trading · 💰 Funded · 📊 Scaling Plan · 💵 Payouts · 👥 Account. Titles are static; values are `data-cd="…"` spans filled by JS on render. (Static HTML defaults exist but are overwritten by `renderCardView()` on load — don't rely on them for data.)

**Full Playbook modal** = numbered `modal-sec` sections (currently 1–6; section 7 "FAQs" was removed). Scrollspy keys off `#mod-N` anchors.

## 2. JS data model — the five objects that hold all per-product data

All live in the first `<script>` block. Editing data = editing these objects (then re-validate + render-test).

### `ACCOUNT_DATA[type].{eval|funded}` → Comparison-view cards (data-driven rows)
- `.fixed` (per phase, type-level) and `.sizes[size]` (per size). The exact fields differ by phase:
  - **eval.fixed:** `{evalSteps, dll, consistency, daysToPass, maxAccounts, resets, activation, news, bots}` · **eval.sizes[size]:** `{target, drawdown, contracts}`
  - **funded.fixed:** `{consistency, daysToPayout, scalingPlan, maxAccounts, resets, split, news, bots, dll}` · **funded.sizes[size]:** `{maxPayout, target, drawdown}`
- Row order & labels live in two ordered schemas: **`EVAL_ROWS`** (12) and **`FUNDED_ROWS`** (12), each an array of `[label, dataKey]`. To add/reorder/relabel a row, edit the schema; to change a value, edit `ACCOUNT_DATA`.
- Consumed by `updateAccountCard(card, size)`: merges `fixed`+`sizes[size]`, picks the schema by `phaseState`, and **sets `.ac-stats` innerHTML** (`<div class="ac-stat-row"><div class="ac-stat-label">…</div><div class="ac-stat-val">…</div></div>`). Missing keys render as an em-dash. No `[data-field]` attributes and no `PHASE_LABELS` anymore — both removed.
- Size pills are **never** rewritten by JS now — they show the static price HTML in both phases (Funded copies Eval). Max payout is a Funded **row**, not a pill.
- **Gotcha:** the drawdown (Total Drawdown) is the same value eval vs funded for a given size — keep them in sync. Rapid 100K = $2,500, Legacy 100K = $3,000 (they diverge at 100K). Eval-only fields: `daysToPass`, `contracts`, `activation`. Funded-only fields: `maxPayout`, `daysToPayout`, `scalingPlan`, `split`.

### `CARD_DATA[type]` → Card-detail view (six boxes, header, about, lists)
- Top-level: `icon, name, tag, tagClass, sub, split, splitFull, targetSub, payoutsTop, minDays, evalMinDays, fundedMinDays, consistencyEval, consistencyFunded, instruments, firstPayout, subsequent, frequency, accountReset, about, features[], bestFor[], notIdeal[], ctaText`.
- `.sizes[size]`: `{fee, target, dll, maxloss, maxlossShort, contracts}`.
- Consumed by `renderCardView()` → `setText(field,val)` writes every `[data-cd=field]`. `setFeeHTML` runs fees through `discountedPrice()`.

### `FUNDED_PAYOUT[type][size]` → Payouts subtab + Funded/Overview payout fields
- Schema: `{firstGoal, sub, maxPayout}` — **per size** (this is what makes the payouts subtab dynamic).
  - `firstGoal` = min for first withdrawal; `sub` = min per subsequent cycle; `maxPayout` = max-per-request label.
- Consumed by `renderPayoutsTab()` (`firstThreshold`/`subsequent`/`maxPerRequest`) and by `renderCardView()` (overview Payouts box `firstPayout`/`subsequent` + `fundedTargetBox`). Also read by `renderFundedTab()` via `fp.firstGoal`.
- **Current maxPayout wording:** Rapid `"$X, no cap after 5"` · Legacy `"Up to $X"` · Bolt `"$1,200 (1st–4), $7,700 (5th)"` · Flex `"Up to 50% · max $X"`.

### `PAYOUT_DATA[type]` + `PAYOUT_ROWS` → top-level Payouts tab (`#panel-payouts`)
- Separate object from the comparison view. `.fixed` (type-level): `{profitTarget, buffer, daysToPayout, consistency, split}`. `.sizes[size]`: `{payoutCap, payoutCapSub, minPayout, drawdown, dll, perDay}`.
- `PAYOUT_ROWS` (11) is the ordered `[label, dataKey]` schema. `renderPayoutCard(account, size)` merges `fixed`+`sizes[size]`, **sets `.ac-stats` innerHTML** from the schema, and writes each size button's `.ac-size-price` to that size's `payoutCap` (so the buttons read as max-payout amounts). Wired on init for every `[data-payout-account]` and on size-button click.
- Note `[data-payout-account]` cards also carry class `account-card`, so `rerenderAllAccountCards()` iterates them too — `updateAccountCard` safely no-ops on them (they have `data-payout-account`, not `data-account`).

### `TAB_CONTENT[type]` → Evaluation / Payouts steps / Scaling / FAQ panels
- `{evalTitle, evalSub, steps[], completion, payouts:{firstThreshold,subsequent,maxPerRequest,steps[]}, hasScaling:false, scalingMsg, faqs[]}`.
- `payouts.firstThreshold/subsequent/maxPerRequest` are **overridden per-size** by `FUNDED_PAYOUT` in `renderPayoutsTab` — so the static values here are fallbacks only.

### `discountedPrice(priceStr)` + `FLEX_LIST`
- Only Flex shows a struck-through base. `FLEX_LIST` maps **sale → base** (currently `{'$79.99':'$133.99','$149.99':'$249.99','$289.99':'$483.99'}`, i.e. 40%-off promo against real published bases). To change the Flex promo, update `FLEX_LIST` keys, the `CARD_DATA.flex.sizes.fee` values, the comparison-card `pp-orig/pp-sale` spans, the comparison-table fee cell, and every hero/sidebar/prose mention together (there are ~16 display spots — grep the old price to find them all).

## 3. Render flow & event wiring
- **Init:** `rerenderAllAccountCards()` (comparison) and `renderCardView()` (card) run on load; the Payouts tab self-initialises per `[data-payout-account]`.
- **Phase toggle** → sets `phaseState`, toggles `.compare-grid.phase-eval`, calls `rerenderAllAccountCards()` + `renderCardView()`. (`rerenderAllAccountCards` just re-runs `updateAccountCard` on each card — the row set/labels switch with `phaseState`.)
- **Type button** → sets `cardState.type`, picks a valid size, `renderCardView()`. Active color via `.cd-type-btn.active[data-cardtype=…]`.
- **Size button** → sets `cardState.size`, `renderCardView()`.
- `renderCardView()` calls the sub-renderers: `renderEvaluationTab`, `renderFundedTab`, `renderPayoutsTab`, `renderScalingTab`, `renderFAQTab`.

### Color coordination (must stay aligned across both views)
`rapid=purple` · `legacy=blue` · `bolt=orange` · `flex=teal`. The active-state CSS selectors that must use these **rapid/legacy/bolt/flex** names (NOT the old growth/select/lightning):
`.cd-type-btn.active[data-cardtype="…"]`, `.cd-size-btn.sel-…`, `.card-detail.type-… .cd-subtab.active`. Comparison cards still carry the legacy family class (`growth/select/lightning/flex`) for their border/pill accents — leave those.

## 4. Per-firm variable map (what to change when cloning to a new firm)
| Slot | Where |
|------|-------|
| Firm name, logo (base64), slogan, CPF score, stars | hero block (`.hero-*`) |
| Rubric rows | `.rubric-row` scores |
| Pros/cons, Allowed/Watch-Out pills | `.pc-list`, `.pill-row` clusters |
| Per-product data | the five JS objects in §2 |
| Comparison table (3-firm) | `.compare-table-real` cells |
| Restricted countries | `#panel-restricted` list |
| Platforms | `PLATFORMS` array + `renderPlatform*` |
| Full playbook | `modal-sec` sections + nav/TOC anchors |
| Promo logic | `FLEX_LIST` / `discountedPrice` (firm-specific) |

**Immutable (do not edit):** color tokens, type system, component CSS, layout grids, spacing scale, the tab/view/toggle machinery. New firms vary in *number of products and sizes* — `.ac-sizes` flex and per-type size-button rebuild already handle 1–4 sizes.

## 5. Verification checklist (run before presenting)
- [ ] `node --check` passes on all `<script>` blocks.
- [ ] Card view: each type button toggles its accent color (purple/blue/orange/teal); six overview boxes populate; switching size updates target/maxloss/contracts/fee and the payouts subtab (first/subsequent/max).

- [ ] Hero score & comparison score = 4.5; slogan correct.
- [ ] Playbook has 6 sections (no `#mod-7`).
- [ ] No console errors (a 403 for a blocked external asset in the sandbox is fine).
